CPU Scheduling

어떤 프로세스를 언제 실행할지 결정하는 정책들을 이해하고, 각 알고리즘의 장단점을 비교해봅니다.

개요 성능 지표 FIFO SJF STCF Round Robin MLFQ 비례 배분
Section 01

CPU 스케줄링이란?

CPU 스케줄링은 실행 가능한(runnable) 프로세스들 중 다음에 어떤 프로세스를 실행할지 결정하는 정책입니다. 프로세스 전환은 매우 자주 일어나므로 알고리즘이 빠르게 동작해야 합니다.

핵심 구분 메커니즘(Mechanism): 어떻게 전환하는가? → 컨텍스트 스위치
정책(Policy): 언제 전환하는가? → 스케줄링 알고리즘

프로세스 상태 전환 다이어그램

프로세스는 Ready → Running → Blocked 사이를 오가며, 스케줄러는 Ready 상태의 프로세스 중 하나를 골라 Running으로 만듭니다.

프로세스 상태 전환 (Ready ↔ Running ↔ Blocked)

주요 용어

Workload

스케줄러에 입력되는 작업 집합 (도착 시간, 실행 시간 등)

Scheduler

언제 어떤 작업을 실행할지 결정하는 로직

Metric

스케줄링 품질을 측정하는 지표 (turnaround, response 등)

Section 02

성능 지표 (Metrics)

스케줄링 알고리즘을 평가할 때는 여러 지표를 사용합니다. 이 지표들은 서로 트레이드오프(trade-off) 관계에 있습니다.

Turnaround Time

작업이 도착 후 완료될 때까지의 시간. 배치 작업에 중요.

T_completion − T_arrival
Response Time

작업이 도착 후 처음 실행될 때까지의 시간. 대화형 작업에 중요.

T_firstrun − T_arrival
Waiting Time

Ready 큐에서 대기하는 총 시간. 짧을수록 좋음.

Throughput

단위 시간당 완료되는 작업 수. 많을수록 좋음.

Fairness

모든 작업이 일정 시간 동안 동등하게 CPU를 받는 정도.

Overhead

컨텍스트 스위치 등 스케줄링 자체에 드는 비용. 낮을수록 좋음.

⚠️ 트레이드오프 Turnaround Time을 줄이면 Response Time이 나빠질 수 있고, Fairness를 높이면 Throughput이 떨어질 수 있습니다. 어떤 지표를 우선시하느냐가 알고리즘 선택의 핵심입니다.
Section 03

FIFO (First-Come, First-Served)

가장 단순한 스케줄링 알고리즘. 먼저 도착한 작업을 먼저 실행합니다. 마트 계산대 줄서기와 동일한 원리입니다.

장점 ✓

구현 간단, 공평(starvation 없음), Non-preemptive

단점 ✗

Convoy Effect: 긴 작업 뒤에 짧은 작업이 끝없이 기다려야 함

예시: 같은 길이의 작업들 (좋은 경우)

A(10), B(10), C(10) — T_turnaround = (10+20+30)/3 = 20

예시: Convoy Effect (나쁜 경우)

A가 100초가 걸리는 큰 작업이고 B, C는 10초짜리인데, A가 먼저 도착해서 B와 C를 오래 기다리게 합니다.

A(100), B(10), C(10) — T_turnaround = (100+110+120)/3 = 110 😱
Section 04

SJF (Shortest Job First)

실행 시간이 가장 짧은 작업을 먼저 실행합니다. 동시에 도착한 상황이라면 최적의 Turnaround Time을 보장함을 수학적으로 증명할 수 있습니다.

💡 왜 SJF가 최적인가? 작업을 짧은 것부터 처리하면, 각 작업의 완료가 다른 작업의 대기 시간에 미치는 영향이 최소화됩니다. 이는 마치 수퍼마켓에서 바구니에 물건이 하나인 사람을 먼저 계산해주는 것과 같습니다.

SJF가 Convoy Effect를 해결한 경우

A(100), B(10), C(10) — B,C를 먼저 → T_turnaround = (10+20+120)/3 = 50

SJF의 한계: 비동시 도착

A가 t=0에 도착하고 B, C가 t=10에 도착한다면? SJF는 Non-preemptive이므로 이미 실행 중인 A를 멈추지 못합니다.

A가 먼저 도착 → B, C는 A 종료 후에야 실행 → T_turnaround = (100+90+100)/3 = 96.7 😟
장점 ✓

동시 도착 시 최적 Turnaround Time

단점 ✗

비동시 도착 시 비최적, Starvation 가능, 실행 시간을 미리 알아야 함

Section 05

STCF (Shortest Time-to-Completion First)

SJF의 선점형(Preemptive) 버전. 새 작업이 도착할 때, 현재 작업의 잔여 시간보다 새 작업의 실행 시간이 짧으면 즉시 선점합니다.

선점(Preemption)이란? 스케줄러가 현재 실행 중인 프로세스를 강제로 중단하고 다른 프로세스를 실행할 수 있는 능력. 더 정교한 스케줄링이 가능해집니다.

SJF vs STCF 비교

위: SJF (T_turnaround=96.7) / 아래: STCF (T_turnaround=50) — B,C 도착 즉시 A를 선점
장점 ✓

비동시 도착 상황에서도 최적 Turnaround Time

단점 ✗

Response Time이 여전히 나쁠 수 있음, 실행 시간을 미리 알아야 함

Section 06

Round Robin (RR)

Ready 큐를 원형 FIFO로 취급합니다. 각 작업에 Time Slice(= 스케줄링 퀀텀)를 부여하고, 슬라이스가 끝나면 다음 작업으로 전환합니다.

💡 Time Slice 크기의 트레이드오프 너무 짧으면 → 컨텍스트 스위치 오버헤드 증가
너무 길면 → 응답 시간(Response Time) 나빠짐
일반적으로 10~100ms 사용

SJF vs RR 비교 (A, B, C 각 30초)

위: SJF — T_response=30, T_turnaround=60 / 아래: RR(10s) — T_response=10, T_turnaround=80
⚠️ Turnaround vs Response 트레이드오프 RR은 Response Time이 훨씬 좋지만, Turnaround Time은 오히려 나쁩니다. 대화형(interactive) 서비스에는 RR이, 배치(batch) 작업에는 SJF/STCF가 유리합니다.
알고리즘 T_turnaround (avg) T_response (avg) Starvation Preemptive
FIFO ✅ 작업 순서에 따름 ❌ 나쁨 없음 No
SJF ✅ 동시 도착 시 최적 ❌ 나쁨 가능 No
STCF ✅ 최적 ❌ 나쁨 가능 Yes
RR ❌ 나쁨 ✅ 좋음 없음 Yes
Section 07

MLFQ (Multi-Level Feedback Queue)

실제 시스템에서는 작업의 실행 시간을 미리 알 수 없습니다. MLFQ는 과거 행동을 관찰해서 미래를 예측하며 우선순위를 동적으로 조정합니다.

🎯 핵심 아이디어 Interactive 작업 (짧고 I/O 많음) → 높은 우선순위 유지
CPU-intensive 작업 (길고 CPU만 씀) → 우선순위 강등

MLFQ의 5가지 규칙

RULE 1
Priority(A) > Priority(B) 이면, A만 실행 (B는 대기)
RULE 2
Priority(A) = Priority(B) 이면, A와 B를 Round Robin으로 실행
RULE 3
새 작업이 시스템에 진입하면 가장 높은 우선순위 큐에 배치
RULE 4 (개선판)
해당 레벨에서 총 time allotment를 다 쓰면 (중간에 I/O를 해도 누적 계산) 우선순위를 한 단계 낮춤
RULE 5 — Priority Boost
주기 S마다 모든 작업을 최상위 큐로 이동 → Starvation 방지 + 행동 변화 반영

MLFQ 구조 시각화

높은 우선순위 큐는 짧은 time slice, 낮은 큐는 긴 time slice

Rule 4 개선: 왜 필요한가?

원래 Rule 4a/4b에서는 time slice가 끝나기 전에 I/O를 호출하면 같은 우선순위를 유지했습니다. 악성 프로그램이 이를 악용해 time slice 직전에 I/O를 호출하면 영원히 높은 우선순위를 유지할 수 있었습니다.

왼쪽: 취약한 방식 — B가 I/O를 남발해 Q2에 머뭄 / 오른쪽: 개선된 방식 — 누적 사용량으로 판단

Priority Boost 시각화

Boost 없이는 A가 굶주림(starvation) / Boost 적용 시 A도 주기적으로 실행
✅ MLFQ의 장점 실행 시간에 대한 사전 지식 없이도 SJF/STCF에 가까운 성능을 달성할 수 있습니다. 실제 Linux, macOS, Windows 모두 MLFQ 변형을 사용합니다.
Section 08

비례 배분 스케줄링 (Proportional Share)

각 프로세스에 가중치(weight)를 부여하고, CPU를 그 비율에 맞게 분배합니다.

🎲 Lottery Scheduling (확률적)

각 작업에 티켓(ticket)을 발급합니다. 전체 M개 중 작업 i가 m_i 개를 가지면, CPU를 받을 확률은:

p_i = m_i / M

스케줄러는 랜덤으로 당첨 티켓을 뽑아 그 작업을 실행합니다. 확률적이므로 단기적으로는 오차가 있지만 장기적으로 수렴합니다.

📐 Stride Scheduling (결정론적)

각 작업에 Stride = 큰 상수 / 티켓 수를 부여하고, Pass 값이 가장 작은 작업을 실행합니다. 실행 후 Pass += Stride.

예시 τ₁: tickets=3 → stride=2  |  τ₂: tickets=2 → stride=3  |  τ₃: tickets=1 → stride=6
time τ₁ pass (stride=2) τ₂ pass (stride=3) τ₃ pass (stride=6) 선택
초기 pass = stride 값. 가장 작은 pass를 가진 작업이 선택됨 (동률은 임의)
Lottery

구현 간단, 새 프로세스 추가 쉬움, 단기적 불확실성 있음

Stride

정확하고 결정론적, 새 프로세스 추가 시 pass 초기화 문제 있음

Linux CFS (Completely Fair Scheduler)

현대 Linux는 Stride 스케줄링의 개선판인 CFS를 사용합니다. vruntime(가상 실행 시간)이 가장 작은 프로세스를 실행하며, 레드-블랙 트리로 O(log n) 시간에 선택합니다.
